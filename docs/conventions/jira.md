# Jira 추적 규칙

이 문서는 Jira 이슈 추적, 브랜치/PR/커밋의 Jira 키 표기, 상태 전환 기준을 정리한다.
필수 규칙과 충돌하면 `AGENTS.md`를 우선한다.

## 기본 기준

- Jira 프로젝트 키는 `PPT`를 사용한다.
- 상태는 `Backlog`, `Selected for Development`, `진행 중`, `검토중`, `완료`를 사용한다.
- 브랜치와 PR은 기능 이슈 단위로 만든다.
- 기능 이슈는 `Epic` 자체가 아니라 `Epic` 바로 아래의 `새 기능(issuetype = 10103)` 또는 기능성 `Task`, `Bug`, `개선` 등을 의미한다.
- 기능 이슈 예시는 `PPT-37 실시간 음성인식`, `PPT-38 음성 기반 슬라이드 제어`처럼 작성한다.
- 커밋은 해당 기능 이슈에 포함되는 `하위 작업` 단위로 나눈다.

## 브랜치, PR, 커밋 표기

- 브랜치명에는 기능 이슈 키를 포함한다.

```text
feature/PPT-37-realtime-stt
```

- PR 제목에는 기능 이슈 키를 포함한다.

```text
[PPT-37] 실시간 음성인식 구현
```

- PR 본문에는 기능 이슈와 포함된 하위 작업을 남긴다.

```text
Parent: PPT-37
Includes: PPT-142, PPT-170, PPT-183
```

- 커밋 본문에는 하위 작업 키와 상위 기능 이슈 키를 남긴다.

```text
Jira: PPT-142
Parent: PPT-37
```

- 브랜치명, PR 제목, PR 본문, 커밋 본문 어디에서도 Jira 키를 찾을 수 없으면 에이전트는 커밋하지 않고 사용자에게 이슈 키를 요청한다.

## 담당자와 상태 변경

- 작업 시작 전 Jira 담당자를 지정한다.
- 담당자가 비어 있으면 에이전트는 임의로 선택하지 않고 사용자에게 확인한다.
- Jira 상태 변경은 사용자가 명시적으로 요청했거나 CI/중앙 자동화가 수행할 때만 한다.
- PR 생성 시 `진행 중` 또는 `Selected for Development` 상태의 기능 이슈를 `검토중`으로 전환한다.
- 테스트 통과만으로 `완료`로 전환하지 않는다.
- `완료` 전환은 PR이 `main`에 병합된 뒤 수행한다.
- CI 실패 시 상태를 되돌리지 않고 Jira 댓글이나 PR 코멘트로 실패 사실만 남긴다.

## 자동화 기준

- 자동 상태 변경 대상은 PR 생성 시 `Parent:` 기능 이슈, 병합 시 `Includes:` 하위 작업 이슈를 기준으로 한다.
- 자동화 구현 시 transition id를 하드코딩하지 않는다.
- Jira API로 가능한 전환을 조회한 뒤 상태 이름으로 매칭한다.
